Part I · How to Attack Any Design Question

Chapter 0 · The 7-Step Design Framework

Foundation Read this first — every later chapter follows this recipe.

Students struggle with system design interviews (SDIs) for three predictable reasons: the questions are open-ended with no single correct answer, most students have never built a truly large system, and — most fixable of all — they simply didn't prepare. At top companies, an above-average SDI performance materially changes both the offer and the level, because it demonstrates that you can handle complexity. The good news: a repeatable 7-step method turns a vague prompt into a structured 35–40 minute conversation.

Step 1 — Requirements clarification

Design questions are deliberately under-specified. Spend the first minutes defining the end goals and choosing which parts of the system you will focus on. For a Twitter-like service you'd ask: Can users post tweets and follow people? Do we build the timeline? Do tweets carry photos and video? Backend only, or front-end too? Is search in scope? Trending topics? Push notifications? Each answer reshapes the design.

Step 2 — System interface definition

Write down the APIs the system must expose. This nails the exact contract and catches requirement misunderstandings early. For example:

postTweet(user_id, tweet_data, tweet_location, user_location, timestamp, ...)
generateTimeline(user_id, current_time, user_location, ...)
markTweetFavorite(user_id, tweet_id, timestamp, ...)

Step 3 — Back-of-the-envelope estimation

Estimate scale before designing: expected traffic (new tweets/sec, timeline views/sec), storage (very different if tweets carry media), and network bandwidth. These numbers drive every later decision about scaling, partitioning, load balancing, and caching.

Step 4 — Defining the data model

Identify the entities, how they interact, and how data flows between components — this guides partitioning and management later. For Twitter: User(UserID, Name, Email, DoB, CreationDate, LastLogin), Tweet(TweetID, Content, TweetLocation, NumberOfLikes, TimeStamp), UserFollow(UserID1, UserID2), FavoriteTweets(UserID, TweetID, TimeStamp). Then ask: NoSQL like Cassandra, or MySQL? What block storage for photos and videos?

Step 5 — High-level design

Draw a block diagram with 5–6 boxes covering the problem end-to-end. For Twitter: multiple application servers behind load balancers; if reads dominate writes, separate read and write server pools; a database built for heavy read volume; distributed file storage for media.

Step 6 — Detailed design

Dig into two or three components — let the interviewer's feedback steer you. Present alternative approaches with pros and cons; there is no single answer, only trade-offs under constraints. Typical deep dives: How do we partition massive data across databases, and what goes wrong if a user's whole dataset sits on one machine? How do we handle "hot" users? Should storage be optimized for scanning the latest tweets? Where do we cache, and how much? Which components need load balancing most?

Step 7 — Identifying and resolving bottlenecks

Hunt for single points of failure and mitigation. Do we have enough data replicas to survive server losses? Enough service copies that a few failures won't cause total shutdown? How do we monitor performance, and do we alert when critical components fail or degrade?

Preparation plus organization wins SDIs. The seven steps keep you on track and guarantee you cover requirements, scale, data, architecture, depth, and reliability — in that order.

Part II · Fifteen Classic Design Problems

Chapter 1 · Designing a URL Shortener (TinyURL)

Easy Similar: bit.ly, goo.gl, qlink.me

1. Why URL shortening?

Short links save space in messages, tweets and print, reduce typos, enable per-link analytics, and can hide the original URL. A long educative.io URL shrinks to something like http://tinyurl.com/jlg8zpc — roughly a third of the size.

2. Requirements

  • Functional: (1) given a URL, generate a shorter, unique alias; (2) redirect visitors of a short link to the original URL; (3) optional custom aliases; (4) links expire after a default timespan, with user-settable expiration.
  • Non-functional: high availability (if we're down, every redirection fails), real-time redirection with minimal latency, and short links must not be guessable.
  • Extended: analytics (redirect counts) and a REST API for other services.

3. Capacity estimation (read-heavy, 100:1 read/write)

Assume 500M new shortenings per month → 50B redirections per month.

Writes: 500M / (30 × 24 × 3600) ≈ 200 URLs/s   |   Reads: 100 × 200 = 20K/s

Storing every object for 5 years: 500M × 5 × 12 = 30B objects × ~500 bytes ≈ 15 TB. Bandwidth: incoming 200 × 500 B = 100 KB/s; outgoing 20K × 500 B ≈ 10 MB/s. Memory for cache: 20K req/s ≈ 1.7B req/day; caching the hottest 20% (80/20 rule) needs 0.2 × 1.7B × 500 B ≈ 170 GB (less in practice — duplicates).

4. System APIs

createURL(api_dev_key, original_url, custom_alias=None, user_name=None, expire_date=None)
  → returns the short URL, or an error code
deleteURL(api_dev_key, url_key) → returns 'URL Removed'

The api_dev_key throttles each registered developer to a quota of creations/redirections per period — this is how we prevent a malicious user from consuming all keys.

5. Database design

Observations: billions of small records (<1K each), no relationships beyond "who created it", read-heavy. Two tables: URL(Hash PK varchar(16), OriginalURL varchar(512), CreationDate, ExpirationDate, UserID) and User(UserID PK, Name, Email, CreationDate, LastLogin). Because we need billions of rows and no joins, a NoSQL key-value store (DynamoDB, Cassandra, Riak) fits best and scales more easily.

6. Key generation — the heart of the problem

Approach A: encode the actual URL

Compute MD5/SHA256 of the URL and encode it in base36/base62/base64. A 6-character base64 key gives 64⁶ ≈ 68.7B strings; 8 characters give 64⁸ ≈ 281T. Six characters suffice. MD5 yields 21+ base64 characters, so we take the first 6–8 — resolving duplicates by swapping/choosing other characters. Problems: identical URLs from different users would collide into the same key, and URL-encoded variants of the same address hash differently. Workarounds: append an increasing sequence number (overflow and performance concerns) or append the user id (signed-out users must pick a uniqueness key; conflicts still need retries).

Approach B: offline Key Generation Service (KGS) — preferred

A standalone KGS pre-generates random 6-letter keys into a key-DB. Shortening just takes one — no encoding, no collisions. Concurrency: KGS keeps two tables (unused/used keys); as soon as keys are loaded into memory it moves them to "used", and it synchronizes access so no two app servers get the same key. If KGS dies with keys in memory they're wasted — acceptable with 68B available. Key-DB size: 6 bytes × 68.7B ≈ 412 GB. KGS is a single point of failure, so run a standby replica. App servers may cache keys locally (dying servers waste them, also fine).

Lookup: hit the datastore; if found, return HTTP 302 Redirect with the original URL in the Location header; else HTTP 404 or redirect home. Custom aliases get a size cap (16 characters).

7. Partitioning and replication

Range-based (by first letter of the hash key) is predictable but unbalances servers — too many URLs may start with 'E'. Hash-based maps the key's hash to 1…256 partitions; hot partitions remain possible and are fixed with Consistent Hashing.

8. Cache, LB, cleanup, telemetry, security

Cache: Memcached holding full URLs by hash; start at 20% of daily traffic (~170 GB, fits one 256 GB server or a few smaller ones). LRU eviction with a Linked-Hash-Map; replicate cache servers, and on a miss update all replicas (a replica already holding the entry ignores it). Load balancers go in three places: client↔app, app↔DB, app↔cache; start with round-robin (removes dead servers, but ignores load) and evolve to load-aware LB. Cleanup: lazy expiry — delete on access if expired, plus a lightweight periodic cleanup service in off-peak hours; default expiry (e.g., two years); reclaimed keys return to key-DB. Telemetry: track visitor country, timestamp, referrer, browser/platform — beware hot rows updated per view. Security: store public/private permission per URL, plus a permissions table keyed by hash listing allowed UserIDs; unauthorized access returns HTTP 401.

Why is 302 (temporary) rather than 301 (permanent) redirect a deliberate choice here? Hint: with 301 the browser caches the mapping — and your analytics never see repeat clicks.

Chapter 2 · Designing Pastebin

Easy Similar: pastebin.com, pasted.co, chopapp.com

1. What it is & requirements

Users paste text and receive a random URL to access it. Functional: upload text and get a unique URL; text only; automatic expiry with user-settable expiration; optional custom alias. Non-functional: high reliability (no data loss), high availability, real-time access, unguessable links. Extended: analytics and a REST API.

Design considerations: cap pastes at 10 MB to stop abuse; cap custom URL length for a consistent database.

2. Capacity estimation (5:1 read/write)

1M new pastes/day, 5M reads/day → 12 writes/s, 58 reads/s. Average paste ≈ 10 KB → 10 GB/day → 36 TB over 10 years. Keys: 3.6B pastes need 6-char base64 keys (64⁶ ≈ 68.7B), costing 3.6B × 6 ≈ 22 GB (negligible). With a 70% capacity model, provision 51.4 TB. Bandwidth: 120 KB/s in, 0.6 MB/s out. Cache 20% of daily reads: 0.2 × 5M × 10 KB ≈ 10 GB.

3. APIs

addPaste(api_dev_key, paste_data, custom_url=None, user_name=None, paste_name=None, expire_date=None)
  → URL of the paste, or error
getPaste(api_dev_key, api_paste_key) → paste text
deletePaste(api_dev_key, api_paste_key) → true/false

4. Storage design

Billions of records; metadata objects are tiny (<100 bytes) but paste contents can reach a few MB — so separate them: a metadata DB (Paste(URLHash PK, ContentKey, ExpirationDate, CreationDate) + User table) and object storage (Amazon S3) for contents. URLHash is the short-URL key; ContentKey points at the stored object. This split lets each layer scale independently.

5. Component design

Writes: the app server generates a random 6-letter key (or uses the user's custom key), stores contents + key; on duplicate-key failure it retries with a new key; duplicate custom keys return an error to the user. Alternatively, reuse the KGS pattern from Chapter 1 — pre-generated unique keys, two-table used/unused bookkeeping, standby replica for the KGS single point of failure, optional per-server key caching. Reads: look up the key; return contents or an error. Purging, partitioning, cache, LB, and security all follow the URL-shortener designs.

Notice the pedagogy: Pastebin is TinyURL plus one twist — payloads big enough to justify splitting metadata from object storage. Interviewers love asking a "variant" problem to see if you can transfer a design.

Chapter 3 · Designing Instagram

Medium Similar: Flickr, Picasa

1. Scope & requirements

We design a simplified photo-sharing service: users upload photos, follow others, and see a News Feed of top photos from people they follow. Functional: upload/download/view photos; search by photo/video title; follow users; generate a News Feed of top photos from followees. Non-functional: high availability; ≤200 ms feed-generation latency; eventual consistency is acceptable (missing a photo briefly is fine); high reliability — no uploaded photo may ever be lost. Out of scope: tags, comments, tagging users, follow suggestions.

Design considerations: read-heavy — optimize fast photo retrieval; storage efficiency matters because uploads are unlimited; low read latency; 100% data durability.

2. Capacity estimation

  • 500M total users, 1M daily active.
  • 2M new photos/day ≈ 23/s, average 200 KB per photo.
  • Daily photo storage: 2M × 200 KB = 400 GB/day; ten years ≈ 1425 TB.

3. High-level design & schema

Two paths — upload and view/search — backed by object storage for image files plus a metadata database. Photo metadata needs an index on (PhotoID, CreationDate) because we always fetch recent photos first. An RDBMS handles the joins but scales painfully; instead: photos in distributed file storage (HDFS or S3), metadata in a key-value store (key = PhotoID → PhotoLocation, UserLocation, CreationTimestamp…), and the two relationship tables in a wide-column store like Cassandra — UserPhoto (UserID → list of PhotoIDs) and UserFollow. Cassandra maintains replicas for reliability, and deletes are retained for some days (supporting undelete) before permanent removal.

4. Data size estimation (10 years)

5. Component design — split reads from writes

Uploads go to disk and are slow; reads (especially cached ones) are fast. Web servers have a connection cap (say 500 concurrent), so heavy uploads can starve reads. Solution: dedicated read servers and write servers, letting each scale and be optimized independently.

6. Reliability and redundancy

Losing files is not an option: keep multiple copies of every file on different storage servers, and multiple replicas of every service. If only one instance of a service must run, keep a hot standby that takes over on failover (automatic or manual). Redundancy removes single points of failure.

7. Data sharding

a) Partition by UserID

Shard = UserID % 10 (3.7 TB / 1 TB shards → 4 needed; keep 10 for headroom). PhotoIDs come from per-shard auto-increment, made global by appending the shard number. Issues: hot (celebrity) users overload one shard; photo counts are non-uniform; a user's data may outgrow one shard; one shard down = that user fully unavailable.

b) Partition by PhotoID — better

Generate a globally unique PhotoID first, then shard = PhotoID % 10. Since auto-increment inside shards can't work now, dedicate a key-generating DB with a single 64-bit ID table. That DB is a single point of failure — fix with two servers, one issuing odd IDs, one even, behind a round-robin load balancer:

KeyGeneratingServer1: auto-increment-increment = 2, auto-increment-offset = 1
KeyGeneratingServer2: auto-increment-increment = 2, auto-increment-offset = 2

They may drift out of sync — harmless. The same pattern extends to Users, Comments, and other entities; or reuse Chapter 1's KGS. For future growth, create many logical partitions mapped onto fewer physical servers via a config file; migrating a partition is just a config change.

8. News Feed generation

Naive: fetch metadata for the latest 100 photos of every followee, run them through a ranking algorithm (recency, likes…), return the top 100 — but multi-table queries plus sort/merge/rank means high latency. Better: pre-generate the feed continuously into a UserNewsFeed table; the feed request becomes a single lookup, and regeneration resumes from the last generated timestamp.

Delivery models: Pull (clients poll — data goes stale between polls and most polls return empty), Push (long-poll delivery on publish — celebrities with millions of followers make pushes explode), Hybrid (push only for users with few followers; heavy-follow users pull; or cap push frequency and let busy accounts pull).

9. Feed with sharded data — creation time inside the ID

Make the PhotoID itself sortable by time: high bits = epoch seconds, low bits = auto-increment sequence. Fifty years of seconds ≈ 86400 × 365 × 50 = 1.6B → 31 bits; at 23 photos/s, 9 bits of sequence (512/s, reset every second) suffice. The primary index on PhotoID then makes "latest photos" queries extremely fast. Shard = PhotoID % 10.

10. Cache & load balancing

Geo-distributed photo cache servers plus CDNs push content close to users. Metadata gets a Memcache layer with LRU eviction. Under the 80/20 rule, cache 20% of daily photo read volume (and metadata).

Chapter 4 · Designing Dropbox / Google Drive

Medium Similar: OneDrive, Google Drive

1. Why cloud storage & requirements

Cloud file storage won because we now use many devices and platforms: it promises availability (any device, anywhere), reliability/durability (multiple geo-distributed copies), and scalability (pay-as-you-grow). Requirements: upload/download from any device; share files/folders with other users; automatic cross-device sync; files up to a GB; ACID guarantees on file operations; offline editing with sync on reconnect. Extended: snapshotting/versioning of files.

2. Design considerations & scale

Huge read and write volume, roughly balanced. The pivotal trick: chunk every file (e.g., 4 MB pieces). Failed transfers retry only the failed chunk; only modified chunks are transferred; duplicate chunks are stored once (saving storage and bandwidth); clients keep a local metadata copy to avoid round trips; small edits can upload diffs rather than whole chunks. Scale: 500M users, 100M DAU, 3 devices each; 100B files at 100 KB average → 10 PB; 1M active connections per minute.

3. High-level architecture

The user designates a workspace folder; every change inside it propagates to the cloud and to all other devices. Components: Block servers (chunk upload/download with clients), Metadata servers (file name, size, directory, sharing info in SQL/NoSQL), and Synchronization servers (notify clients of changes).

4. The client — four parts

  1. Internal Metadata Database — tracks files, chunks, versions, and locations locally.
  2. Chunker — splits files into 4 MB chunks, reconstructs them, detects which parts changed so only those transfer. Optimal chunk size trades off storage-device IOPS, network bandwidth, and average file size.
  3. Watcher — monitors the workspace, notifies the Indexer of create/delete/update, and listens for broadcasts from the Synchronization Service.
  4. Indexer — updates the internal DB with chunk info; after successful upload/download it tells the Synchronization Service to broadcast changes and update remote metadata.

How do clients learn of remote changes? Periodic polling wastes bandwidth and delays updates; use HTTP long polling — the server holds the request open until there's news. Clients should back off exponentially when servers are slow; mobile clients sync on demand to save bandwidth and battery.

5. Metadata DB, Sync Service, Message Queues, Block storage

Metadata DB (MySQL or DynamoDB) stores chunks, files, users, devices, workspaces — and must give the Synchronization Service a consistent view when several users edit one file. A relational DB gives ACID natively; with NoSQL, the sync logic must supply ACID semantics itself. Synchronization Service — the heart of the system — applies each client's updates to all subscribed clients and reconciles local DBs with remote metadata; offline clients poll on reconnect. It transmits only differences, using SHA-256 hashes per chunk to decide whether a copy is needed; if the server already holds a chunk with the same hash (even from another user!), it just references it. Message Queuing Service decouples everything: one shared, global Request Queue for incoming metadata updates, and a per-client Response Queue for delivering change notifications (messages are deleted once received, so each subscriber needs its own queue). Cloud/Block Storage holds the chunks; clients talk to it directly, and separating metadata from storage lets us swap providers.

6. File processing workflow

  1. Client A uploads chunks to cloud storage.
  2. Client A updates metadata and commits the change.
  3. Client A receives confirmation; notifications go to clients B and C.
  4. B and C receive the metadata change and download the updated chunks (offline clients get theirs from their response queues later).

7. Data deduplication

Hash each incoming chunk and compare against existing hashes. Post-process dedupe stores first, dedupes later — no upload latency, but duplicate data is briefly stored and fully transferred. In-line dedupe hashes in real time on the client; if the chunk exists, store only a reference — optimal network and storage use.

8. Metadata partitioning

Vertical (tables-per-feature on separate servers): simple, but giant tables still need further splitting and cross-DB joins hurt. Range-based (by first letter of file path): predictable but unbalanced. Hash-based (hash of FileID → partition 1…256): uniform, with residual hotspots solved by Consistent Hashing.

9. Cache, LB, security

A Memcached-style block cache serves hot chunks (a 144 GB server caches ~36K chunks at 4 MB); LRU eviction; plus a metadata cache. Load balancers sit between clients↔block servers and clients↔metadata servers — round robin first, load-aware later. For privacy, store per-file permissions in the metadata DB governing visibility and modification, since users can share files or make them public.

Chunking + hashing is the engine of this whole design: it powers partial sync, resumable transfers, diff uploads, and cross-user deduplication with one mechanism.

Chapter 5 · Designing Facebook Messenger

Medium Real-time one-on-one chat, web + mobile

1. Requirements

  • Functional: one-on-one conversations; track online/offline status; persist chat history.
  • Non-functional: real-time, minimal latency; high consistency — identical history on every device; availability is desirable but may be traded for consistency.
  • Extended: group chats; push notifications for offline users.

2. Capacity estimation

500M DAU × 40 messages/day = 20B messages/day. At 100 bytes each: 2 TB/day; five years ≈ 3.6 PB (before compression/replication, plus metadata). Bandwidth: 2 TB / 86400 s ≈ 25 MB/s in and 25 MB/s out (every message leaves again).

3. High-level design & message flow

A chat server orchestrates everything:

  1. User-A sends a message for User-B to the chat server.
  2. Server acks to User-A.
  3. Server stores the message and delivers it to User-B.
  4. User-B acks to the server.
  5. Server notifies User-A of successful delivery.

4. Message handling — push beats pull

With pull, clients poll for new messages: the server must track undelivered messages and most polls return empty — wasteful and slow. With push, active users keep a connection open and the server relays instantly, with no pending-message bookkeeping. Keep connections open via HTTP long polling (or WebSockets): the server holds each request until data arrives; timeouts trigger immediate reconnects.

The server maps users to connections with a hash table (UserID → connection object). If a receiver has gone offline: notify the sender to retry (retry logic lives in the client so nobody retypes), or store briefly and retry on reconnect. Capacity: 500M concurrent connections at 50K per server → 10K chat servers; a software load balancer maps each UserID to its server.

Delivering a message = store it, deliver to receiver, ack the sender — and the ack shouldn't wait for storage (write in the background). Ordering: server timestamps can't give a true order (if M1 arrives at T1 and M2 at T2>T1 from opposite directions, the two users would still see different orders). Keep a per-user sequence number: each client sees a consistent order on all its devices, even though the two participants' views may differ.

5. Storage — why HBase

We need a very high rate of tiny inserts and fast range scans over recent messages. RDBMS or MongoDB would incur a disk read/write per message — high latency and crushing load. A wide-column store like HBase (modeled on BigTable, atop HDFS) batches writes in a memory buffer and flushes to disk when full — perfect for lots of small, variably-sized data and range reads. Clients should paginate fetches, sized to the viewport (phones show fewer conversations). Options for storing on receipt: a separate worker thread, or an async DB request — while designing, mind connection pooling, retrying failures, logging unrecoverable failures, and replaying them later.

6. Managing user status

The connection table already tells us who's online. Broadcasting every flap to 500M users is too expensive, so: pull friends' statuses at app start; report failure when messaging someone offline and update the client; delay "came online" broadcasts a few seconds (they may vanish again); pull statuses only for users in the viewport, infrequently; refresh status when opening a new chat.

7. Partitioning, cache, LB, fault tolerance

Partition by UserID hash so a user's whole history sits on one shard: 3.6 PB / 4 TB ≈ 900 → round to 1K shards, hash(UserID) % 1000; fetching a conversation range is fast. Start with many logical partitions on few physical servers. Partitioning by MessageID would scatter a conversation — ruled out. Cache the last ~15 messages of the ~5 visible conversations, co-located on the user's shard. LB in front of chat servers (UserID → connection holder) and cache servers. Failures: failing over live TCP connections is extremely hard — let clients reconnect automatically. Never keep one copy of messages: replicate or use Reed-Solomon encoding.

8. Extended requirements

Group chat: GroupChatID-keyed objects on chat servers holding member lists; the LB routes by GroupChatID and the handling server fans out to each member's connection server; a separate table partitioned by GroupChatID stores them. Push notifications: a Notification server relays messages for offline users to each device manufacturer's push servers (opt-in per device).

Messenger is the one design in this course that deliberately chooses consistency over availability — compare with Instagram and Twitter, which choose the opposite. Be ready to defend that choice with CAP vocabulary (Part III).

Chapter 6 · Designing Twitter

Medium Post tweets, follow people, favorite tweets

1. Requirements

  • Functional: post tweets (140 chars); follow users; favorite tweets; generate a timeline of top tweets from followees; tweets may carry photos and videos.
  • Non-functional: high availability; timeline generation ≤ 200 ms; consistency can take a hit — briefly missing a tweet is fine.
  • Extended: search, replies, trending topics, tagging users, notifications, follow suggestions, Moments.

2. Capacity estimation

1B total users, 200M DAU, 100M new tweets/day, average 200 followees per user. Favorites: 200M × 5/day = 1B/day. Tweet-views: each DAU visits their timeline twice and five other pages, 20 tweets per page → 200M × 7 × 20 = 28B views/day.

Storage: 140 chars × 2 bytes + 30 bytes metadata ≈ 310 B/tweet → 100M × 310 ≈ 30 GB/day of text. Media: every 5th tweet a 200 KB photo, every 10th a 2 MB video → (100M/5 × 200 KB) + (100M/10 × 2 MB) ≈ 24 TB/day.

Bandwidth: ingress 24 TB/day ≈ 290 MB/s. Egress (28B views; every photo shown; every 3rd video watched): text 93 MB/s + photos 13 GB/s + videos 22 GB/s ≈ 35 GB/s total.

Writes ≈ 1,150 tweets/s average · Reads ≈ 325K tweets/s average — peaks: several thousand writes/s, ~1M reads/s

3. API & high-level design

tweet(api_dev_key, tweet_data, tweet_location, user_location, media_ids, maximum_results_to_return)
  → URL of the posted tweet, or HTTP error

Media upload is separate (via media_ids). Architecture: app servers behind load balancers; a database efficient for enormous read volume; file storage for photos/videos. Schema: Tweet(TweetID, UserID, Content, TweetLatitude/Longitude, UserLatitude/Longitude, CreationDate, NumFavorites), plus User, UserFollow, and Favorite tables (SQL vs NoSQL trade-offs as in Instagram).

4. Data sharding — the central discussion

By UserID

hash(UserID) → server holding all the user's tweets/follows/favorites. Problems: hot celebrity users; non-uniform growth. Fixes require repartitioning or consistent hashing.

By TweetID

Each tweet lands on a random server. Timeline generation: app server finds followees → queries all servers → each returns its matching tweets sorted by recency → aggregator merges. Solves hot users, but every read fans out to every partition — higher latency (mitigated by caching hot tweets).

By creation time

Fetching the latest tweets touches few servers, but both write and read load pile onto the server holding the newest data while old servers idle. Rejected alone.

Combined: TweetID that encodes creation time — the winning design

Make TweetID = [epoch seconds | auto-increment sequence]. 50 years of seconds ≈ 1.6B → 31 bits; 1,150 tweets/s needs 17 bits of sequence (130K/s, reset each second) → a 48-bit ID. Two key-generating DB servers (odd/even IDs) give fault tolerance. Example IDs from epoch 1483228800: 1483228800-000001, 1483228800-000002, … A 64-bit ID would cover 100 years at millisecond granularity. We still fan out reads to all servers, but there's no secondary index on time (faster writes) and no creation-time filtering (the primary key embeds it).

5. Cache

Memcache stores whole tweet objects; LRU eviction. Smarter: with the 80/20 rule, cache 20% of daily read volume per shard. Smarter still: if 80% of users only look at the last 3 days of tweets, cache all tweets from the last three days — under 100 GB of text, one server, replicated for read throughput (do the same for 3 days of hot media). Cache structure: hash table with OwnerID → doubly-linked list of that user's recent tweets, newest inserted at the head, evicted from the tail.

6. Replication, LB, monitoring

Read-heavy → several read-only secondaries per partition; writes go to the primary and replicate out; failover to a secondary on primary death. LB layers: clients→app servers, app→DB replicas, aggregators→cache — round robin first, load-aware later. Monitor: new tweets/day and per-second peaks, timeline delivery rates, and user-perceived timeline refresh latency — these tell you when to add replication, LB, or cache.

7. Extended requirements

Feed serving: merge/sort latest tweets from followees, paginate, fetch top N sized to viewport, pre-cache the next page; or pre-generate feeds (Instagram, Chapter 3). Retweet: store only the original tweet's ID in the new tweet object — no content copy. Trending topics: cache the most frequent hashtags/queries over the last N seconds, refresh every M seconds; weight by tweets/searches/retweets/likes and audience size. Who to follow: suggest friends-of-followees two or three hops out, prefer high-follower accounts, and use ML to shuffle — signals include rising follower counts, mutual followers, location, interests. Moments: mine top news per website over 1–2 hours, find related tweets, classify with supervised learning or clustering. Search: Chapter 10.

Read:write is 200:1 — segregate read from write traffic. Metadata uses master-slave: writes hit the master, reads spread over slaves; brief staleness (milliseconds) after a new upload is acceptable. Uploads: support resume after disconnection — videos are huge. Encoding: queue-driven; notify the uploader when done.

6. Metadata sharding

By UserID: hot users, uneven growth, and title searches must fan out to all servers anyway (aggregate + rank at a central server). By VideoID: hash each VideoID to a server; searches still query all servers, but hot users stop being a problem — hot videos remain one, fixed with a hot-video cache in front of the DBs.

Cache: geo-distributed cache servers; Memcache for metadata rows, LRU; cache 20% of daily read volume by the 80/20 rule. CDN: replicate popular videos at edge locations that serve mostly from memory; long-tail videos (1–20 views/day) stay in our data centers. Fault tolerance: consistent hashing also handles dead database servers and load redistribution.

Chapter 8 · Designing Typeahead Suggestion

Medium Real-time search autocomplete

1. Requirements

As the user types, suggest the top 10 terms beginning with the current prefix — not to make search faster, but to help users articulate queries. Suggestions must appear in real time: ≤ 200 ms.

2. The core data structure: a Trie

We must store a huge set of strings queryable by any prefix — too latency-critical for a database, so the index lives in memory in a trie: a tree where each node holds one character and each root-to-node path spells a prefix (store 'cap', 'cat', 'caption', 'captain', 'capital', and typing "cap" walks to the 'p' node whose subtree contains all completions). Merge single-branch chains of nodes to save space. Assume case-insensitive data for simplicity.

Top-10 ranking: store, with each terminal node, how many searches ended there (CAPTAIN 100 vs CAPTION 500 tells us what to suggest first). But traversing a whole subtree per keystroke is far too slow — the phrase "system design interview questions" is 30 levels deep. So precompute the top 10 suggestions at every node, trading a big storage increase for speed; save space by storing references to terminal nodes (plus frequencies) instead of full strings, walking parent pointers to reconstruct text. Build the trie bottom-up: each node computes its top list from its children's lists.

3. Updating the trie

5B searches/day ≈ 60K QPS — updating per query would crush reads. Instead: log queries (sample if needed — logging every 1000th query is safe when terms below 1000 hits aren't shown), run hourly MapReduce jobs to compute frequencies, then update offline. Two rollout options: each server updates a copy of its trie and atomically switches; or a master-slave pair per trie server — update the slave, promote it, then refresh the old master. Frequencies update as differences, weighted by Exponential Moving Average (recent data counts more). After inserting a term at its terminal node, walk up to the root: at each ancestor, if the term is in the top 10, update its frequency; otherwise insert it if it now qualifies, evicting the lowest. Removals (legal, hate, piracy) happen at the regular update, with an interim filtering layer on each server. Ranking can also weigh freshness, location, language, demographics, and personal history.

4. Persisting the trie

Snapshot periodically so a restarted server can rebuild. Store level-by-level: each node saves its character and child count, followed by its children — e.g., "C2,A2,R1,T,P,O1,D". Top-suggestion lists aren't stored (children don't exist yet when a parent is written); they're recomputed during the rebuild, each node passing its list up to its parent.

5. Scale estimation & partitioning

Of 5B daily queries assume 20% unique, index only the top 50% → 100M unique terms. Average query: 3 words × 5 chars = 15 chars ≈ 30 bytes → 3 GB, plus 2% daily growth kept for a year ≈ 25 GB. Partitioning schemes:

  • Range-based (first letter): predictable, but statically defined letter ranges always end up unbalanced.
  • By server capacity: fill server 1 until memory runs out (say 'A'–'AABC'), server 2 continues ('AABD'–'BXA'), etc., with a lookup map in the load balancer. Typing 'A' queries servers 1+2; 'AAA' only server 1. Merging happens server-side via an aggregator layer. Hotspots (everyone typing "cap") persist.
  • By hash of the term: uniform and hotspot-free, but every query asks all servers and aggregates.

6. Cache, replication, fault tolerance, client tricks

A small fraction of terms dominates traffic — cache top searches (optionally with a small ML model predicting engagement from counts, personalization, trends) in front of trie servers. Replicate trie servers for load and fault tolerance; on a crash, the slave takes over and rebuilding servers load the last snapshot. Client-side optimizations: only fire after 50 ms of no typing; cancel in-flight requests while the user keeps typing; wait for the first couple of characters; prefetch likely data; cache recent suggestion history locally (high reuse); open the server connection as soon as the search page loads (connection setup is a huge share of perceived latency); push parts of the cache to CDNs and ISPs. Personalization: store per-user history server-side, cache it on the client, and rank personal matches above global ones.

Chapter 9 · Designing an API Rate Limiter

Medium Throttle users by request volume

1. What & why

A rate limiter caps how many events an entity (user, device, IP) may perform in a time window — e.g., one message per second, three failed card transactions per day, twenty accounts per IP per day — and blocks requests past the cap. It defends against application-layer abuse: DoS floods, brute-force password and credit-card attempts (bot traffic that looks human and is hard to detect), plus misbehaving or sloppy clients, 2FA-attempt limiting, cost and resource control, tier-based revenue models (buy higher limits), spam prevention, and smoothing traffic spikes.

2. Requirements

  • Functional: limit requests per entity per window (e.g., 15 req/s), enforced across the whole cluster, with an error once the threshold is crossed.
  • Non-functional: highly available (it protects us from attack) and adding minimal latency.

When throttled, the server returns HTTP 429 — Too Many Requests. Throttling flavors: hard (never exceed), soft (exceed by a set percentage, e.g. 100 + 10% = 110 msgs/min), and elastic/dynamic (exceed while spare capacity exists).

3. Fixed Window vs. Rolling Window

Fixed window: windows align to clock boundaries (0–60 s); with a limit of 2 msgs/s, of messages m1,m2 in second one and m3,m4,m5 in second two, only m5 throttles. Rolling window: the window starts at each request's timestamp; the same trace throttles m3 and m4. Fixed windows permit bursts of up to double the rate at boundaries (3 requests in a minute's last second + 3 in the next minute's first second = 6 in ~2 s).

4. Fixed-window design & its two problems

Keep a hash table: UserID → {Count, StartTime}. On each request (limit 3/min): if the user is absent, insert {1, now} and allow; if now − StartTime ≥ 1 min, reset {1, now} and allow; otherwise allow and increment if Count < 3, reject if Count ≥ 3.

  1. Boundary bursts — fixed windows allow up to 2× the rate as shown above; the fix is a sliding window.
  2. Atomicity — in a distributed setup, two processes may read Count=2 concurrently and both allow — a race. Redis locks around read-update solve it (slowing concurrent same-user requests); Memcached has comparable issues; a custom store can lock per record.

Memory: UserID 8 B + Count 2 B (counts to 65K) + minute:second timestamp 2 B = 12 B; +20 B hash overhead → 32 B/user → 32 MB for 1M users (36 MB with a 4-byte lock). Fits one server — but at 10 req/s per user that's 10M QPS of limiter traffic: use distributed Redis/Memcached; app servers read-and-update the remote store before serving or throttling.

5. Sliding Window (exact)

Store per user a Redis Sorted Set of request timestamps. On each request: evict timestamps older than now − window; reject if the set's size ≥ limit; otherwise insert now and accept. Memory explodes: for 500 req/hour: 8 + (4 B time + 20 B sorted-set overhead) × 500 + 20 B hash overhead ≈ 12 KB/user → 12 GB for 1M users — roughly 375× the fixed window. (The 20-byte per-element overhead ≈ two 8-byte neighbor pointers + 4 bytes slack.)

6. Sliding Window with Counters — the hybrid

Keep one counter per small fixed window (e.g., per minute for an hourly limit — 1/60th of the window); a request is throttled when the sum of the last hour's counters exceeds the limit (optionally with a secondary per-minute cap, e.g. 500/hour and 10/min — real users retry successfully since minute counters reset). Store counters in a Redis Hash (very compact under 100 keys), incrementing and setting a 1-hour expiry per request, normalizing times to the minute. Memory: 8 + (4 + 2 + 20 Redis-hash overhead) × 60 + 20 ≈ 1.6 KB/user → 1.6 GB for 1M users — 86% less than the exact sliding window.

7. Sharding, caching, and identity

Shard by UserID with consistent hashing for replication/fault tolerance; different APIs can get different limits (separate limiters for createURL vs deleteURL), and partitioned APIs can carry per-shard sub-limits (e.g., 3 short-URLs/min per partition within 100/hour). Cache recent active users; a write-back cache (update counters in cache, flush to storage at intervals) keeps limiter latency minimal — reads always hit the cache first, which is ideal once a user is maxed out and we're only reading. LRU eviction.

Throttle by IP or by user? IP-based punishes everyone behind a shared IP (cafés, carrier NAT) and IPv6 gives attackers practically unlimited addresses to exhaust your memory. User-based needs authentication — but then the login API itself must be limited, and an attacker who spams wrong passwords can lock out the real user (a DoS). Hybrid per-IP and per-user limiting covers both weaknesses at the cost of more cache entries and memory.

Chapter 11 · Designing a Web Crawler

Hard a.k.a. spiders, robots, worms, walkers, bots

1. What & requirements

A crawler browses the Web methodically, collecting documents by recursively following links from seed pages — powering search indexes, page/link validation, change monitoring, mirror maintenance, copyright search, and special-purpose indexes (e.g., multimedia understanding). Requirements: scalability (crawl the entire Web — hundreds of millions of documents) and extensibility (modular; new document types and protocols pluggable later).

Scoping questions: HTML only, or images/videos too (each media type gets its own parsing module)? HTTP only, or FTP as well (assume HTTP, extensible)? Scale: assume 1B websites → upper bound 15B pages. Courteous crawlers implement the Robots Exclusion Protocol, fetching robots.txt before any real content.

2. Capacity estimation

15B pages / (4 weeks × 7 × 86400 s) ≈ 6,200 pages/s

Storage: 100 KB/page + 500 B metadata → 15B × 100.5 KB ≈ 1.5 PB; at 70% fill → 2.14 PB.

3. The crawl loop & difficulties

  1. Take a URL from the unvisited list → 2. resolve its IP → 3. connect and download → 4. parse for new URLs → 5. add new URLs to the unvisited list → 6. process/store/index the document → repeat.

BFS is standard; DFS is used opportunistically to reuse a warm connection within one site. Path-ascending crawling (for seed http://foo.com/a/b/page.html also try /a/b/, /a/, /) finds isolated resources with no inbound links. Two facts make crawling hard: the Web's sheer volume (you can only fetch a fraction — prioritize wisely) and its rate of change (pages change while you crawl).

4. Component design

Minimum components: URL frontier, HTTP fetcher, extractor, duplicate eliminator, datastore. Worker threads loop: take a URL from the frontier; the scheme (HTTP) picks the protocol module; the document lands in a Document Input Stream (DIS) so several modules can re-read it (≤64 KB cached in memory, larger spilled to a backing file); run the document dedupe test; process by MIME type (modular); the HTML module extracts links; each link is made absolute, passed through the URL filter, then the URL-seen test; new URLs enter the frontier.

Key components in depth

  • URL frontier: distributed across servers, each with many worker threads and one FIFO sub-queue per thread. Politeness: hash each URL's canonical hostname to a fixed thread — so at most one worker connects to a given web server, and FIFO order prevents overload. Frontier size (hundreds of millions) exceeds memory: buffer enqueues to disk, keep a dequeue cache refilled from disk.
  • Fetcher: downloads via protocol modules; caches robots.txt rules per host to avoid refetching.
  • Document dedupe: 64-bit checksum (MD5/SHA) per document; 15B × 8 B = 120 GB — fits memory on a big server, else LRU cache backed by persistent storage.
  • URL dedupe: store fixed-size checksums (not text): 15B × 4 B = 60 GB; an in-memory cache of popular URLs gets a high hit rate. Bloom filters are a compact alternative but false positives mean a URL is never crawled — mitigate with a larger bit vector.
  • URL filters: blacklist by domain, prefix, or protocol before frontier insertion.
  • DNS: a major bottleneck at this scale — run a local caching DNS server.
  • Checkpointing: full crawls take weeks — snapshot state to disk regularly and resume from the latest checkpoint after interruption.

5. Fault tolerance & partitioning

Distribute crawl servers with consistent hashing (dead hosts replaced, load redistributed). Partition by hostname: each host keeps its URLs-to-visit, seen-URL checksums, and document checksums together, checkpointing snapshots to a remote server so a replacement can take over.

6. Crawler traps

Traps are URL sets that make a crawler loop forever — unintentional (filesystem symlink cycles) or intentional (dynamically generated infinite document webs); anti-spam traps catch email-harvesting bots, and some sites trap search crawlers to inflate rankings. Beware also spam sites and cloaked content.

Chapter 12 · Designing Facebook's Newsfeed

Hard Similar: Twitter/Instagram/Quora feeds

1. Requirements

  • Functional: feed built from posts of followed people, pages, and groups; users may have many friends and follow many entities; items may be text, images, or video; new posts are appended to active users' feeds as they arrive.
  • Non-functional: feed generated in real time — ≤2 s end-user latency; a new post reaches a user's feed within ≤5 s of a feed request.

2. Capacity estimation

Average user: 300 friends + 200 followed pages. 300M DAU × 5 fetches/day = 1.5B feed requests/day ≈ 17,500 req/s. Keeping ~500 posts/user in memory at ~1 KB each = 500 KB/user → 150 TB for all active users → at 100 GB/server, ~1,500 machines.

3. API & data model

getUserFeed(api_dev_key, user_id, since_id, count, max_id, exclude_replies) → JSON feed items
  count ≤ 200 per request; since_id / max_id paginate by item ID

Primary objects: User, Entity (page/group), FeedItem (post with UserID; optionally EntityID for where it was posted). Relations: User↔Entity follows (a UserFollow table whose Type column says user-or-entity) and FeedItem↔Media (FeedMedia table).

4. Feed generation

Naive query — select FeedItems whose UserID or EntityID appears in the user's follow list, ordered by CreationDate, limit 100 — has four problems: painfully slow for heavy followers (huge sort/merge/rank); it runs at page-load time (high latency); live updates cause a backlog of feed regenerations; and pushing updates for high-follower accounts creates crushing load.

Offline generation: dedicated servers continuously build feeds and keep them in memory, resuming each time from the timestamp of the last generation. Store per user a hash-table entry:

Struct {
    LinkedHashMap<FeedItemID, FeedItem> feedItems;
    DateTime lastGenerated;
}

A LinkedHashMap/TreeMap lets us jump to the client's last-seen FeedItemID and return the next page. Store ~500 items/user initially (tune down to ~200 if users rarely read past 10 pages of 20); deeper requests hit backend servers. Don't materialize feeds for dormant users: LRU-evict long-inactive users, or learn login patterns and pre-generate just before their usual times.

5. Feed publishing — fanout

Pushing a post to followers is fanout: push = fanout-on-write, pull = fanout-on-load.

  • Pull: keep recent feeds in memory, clients fetch on demand — new data waits for the next pull, and the right cadence is hard (most polls come back empty).
  • Push: push each new post to all followers immediately (long-poll delivery); fetching no longer walks the friend list — big read savings — but celebrity users with millions of followers make writes explosive.
  • Hybrid — the answer: disable fanout for high-follower users (their followers pull); push only for accounts with hundreds/thousands of followers; optionally fan out only to online friends. "Push to notify, pull to serve" combines the best of both.

Cap items per response (~20) but let clients choose (mobile fetches fewer). On mobile, where data is costly, consider not pushing — offer "pull to refresh."

6. Ranking & partitioning

Chronological is simplest, but modern ranking selects signals — likes, comments, shares, update time, media presence — and combines them into a score, then keeps improving by measuring user stickiness, retention, and ads revenue. Partition posts/metadata like Twitter (Chapter 6). Partition in-memory feed data by UserID hash — a user's ≤500 items always fit one server, so one lookup serves the feed; use consistent hashing for growth and replication.

Chapter 13 · Designing Yelp / Nearby Friends

Hard Proximity search over places

1. Requirements & scale

  • Functional: add/delete/update places; given a lat/long, find all nearby places within a radius; add reviews with pictures, text, and a rating.
  • Non-functional: real-time search with minimal latency; searches vastly outnumber place additions.

Scale: 500M places, 100K QPS, ~20% annual growth in both. Crucially, place locations rarely change — a design for frequently-moving objects (people, taxis) would look very different (see Chapter 14).

2. Data model & API

Location: LocationID 8 B (4 would fit 500M, but plan for growth) + Name 256 + Lat 8 + Long 8 + Description 512 + Category 1 = 793 bytes. Reviews: LocationID 8 + ReviewID 4 + ReviewText 512 + Rating 1 (stars out of ten); photos in a similar separate table.

search(api_dev_key, search_terms, user_location, radius_filter, maximum_results_to_return,
       category_filter, sort, page_token)
  sort: 0 = best match, 1 = min distance, 2 = highest rated
  → JSON: name, address, category, rating, thumbnail per result

3. From SQL to grids to QuadTree

a) Plain SQL

Index latitude and longitude separately, then WHERE Latitude BETWEEN X-D AND X+D AND Longitude BETWEEN Y-D AND Y+D (approximating true distance). Each index alone returns an enormous list, and intersecting two huge lists is slow.

b) Fixed grids

Divide the map into cells sized to the usual query radius (e.g., 10 miles); then any search touches only the containing cell plus its eight neighbors — all computable statically from lat/long. Store GridID (4 B) with each place and index it; better yet, keep the whole grid index in memory as a hash of GridID → place list. Earth ≈ 200M mi² → 20M cells: (4 × 20M) + (8 × 500M) ≈ 4 GB. Weakness: places aren't uniform — downtown cells bulge while ocean cells sit empty.

c) Dynamic grids — the QuadTree

Cap each cell at 500 places; when it overflows, split into four equal children. The result is a QuadTree: leaves hold place lists; dense San Francisco gets many small cells, the Pacific a few huge ones. Build: start from one world-node and split recursively. Search: descend from the root to the leaf containing the query point; if it lacks enough results, expand to neighbors — via a doubly linked list threaded through the leaves, or via parent pointers to reach siblings — until you have enough places or exhaust the radius. Memory: cache LocationID + lat/long = 24 B × 500M = 12 GB; 500M/500 = 1M leaves; internal nodes ≈ ⅓ × 1M, each with 4 × 8 B child pointers ≈ 10 MB — ≈12.01 GB total, one server. Inserting a new place touches the DB and the QuadTree (find the right server first if distributed).

4. Partitioning the QuadTree

  • By region (e.g., zip codes): hot regions overload their server and regions grow unevenly — repartition or consistent-hash to recover.
  • By LocationID (chosen): hash each place to a server; each server builds a QuadTree over its subset. Queries fan out to all partitions, which each return nearby places; an aggregator merges. Trees differ across partitions — harmless, since we search neighboring grids on all partitions, and places stay roughly balanced.

5. Replication & fault tolerance

Master-slave QuadTree replicas: reads from slaves (milliseconds staleness OK), writes through the master. Losing primary + secondary requires a rebuild without knowing what the server held — instead of brute-forcing the DB, keep a QuadTree Index server: a reverse map of partition → HashSet of {LocationID, lat/long}, so a rebuilding server just asks for its list (HashSet makes add/remove fast). Replicate this index server too; it can itself be rebuilt by scanning the database.

6. Cache, LB, ranking

Memcache for hot places in front of the DB, LRU, sized by usage. LBs at clients↔app and app↔backend, round robin then load-aware. Ranking by popularity: store an aggregate rating (stars out of ten) in both DB and QuadTree; each partition returns its top 100 by popularity and the aggregator picks the global top 100. Because the design assumes infrequent updates, propagate popularity changes lazily — once or twice a day at low-traffic hours.

Chapter 14 · Designing the Uber Backend

Hard Prerequisite: Chapter 13 (Yelp) · Similar: Lyft, Didi, Via, Sidecar

1. Requirements

Two user kinds — drivers and customers. Drivers continuously report location and availability; customers see all nearby available drivers; a ride request notifies nearby drivers; once matched, both parties see each other's live location until trip end; on arrival, the driver marks the journey complete and becomes available again.

Scale: 300M customers, 1M drivers; 1M daily active customers, 500K daily active drivers; 1M daily rides; all active drivers report location every 3 seconds; ride requests must reach drivers in near real time.

2. Why Yelp's QuadTree isn't enough

The Yelp QuadTree assumed rarely-moving data. Here, every driver moves constantly: updating the tree per report means finding the driver's old grid, removing, reinserting into the new grid, and possibly repartitioning — far too expensive at this update rate. We also need fast propagation of driver positions to watching customers and to both parties during a trip.

3. DriverLocationHT + lazy QuadTree refresh

Keep the latest positions in a hash table — DriverLocationHT — and guarantee the QuadTree reflects a driver's position within 15 seconds (refreshing roughly every 10 s) instead of every 3 s. Record: DriverID 3 B (1M drivers) + old lat 8 + old long 8 + new lat 8 + new long 8 = 35 B → 1M × 35 = 35 MB. Update bandwidth: (3 + 16) B × 1M every 3 s ≈ 19 MB per 3 s. One server could hold it, but distribute by DriverID (for scalability, performance, fault tolerance) onto Driver Location servers, which do two jobs: broadcast each update to interested customers, and notify the appropriate QuadTree server on the ~10 s cadence.

4. Broadcasting to customers — push vs. pull

Push model: a Notification Service on a publish/subscribe model. When a customer opens the app, the server returns nearby drivers and subscribes the customer to those drivers' updates; every DriverLocationHT change is broadcast to subscribers. Memory: 500K daily-active drivers × 5 subscribers, DriverID 3 B + CustomerID 8 B → (500K × 3) + (500K × 5 × 8) ≈ 21 MB. Bandwidth: 2.5M subscribers × 19 B/s ≈ 47.5 MB/s. Implement with long polling or push notifications. Weakness: newly arrived drivers in the customer's viewport require tracking what area each customer watches — complex.

Pull model (simpler, chosen): clients send their location every ~5 seconds and the server queries the QuadTree for nearby drivers, returning fresh positions each time.

Also: let grids grow/shrink an extra 10% beyond the 500-place limit before splitting/merging, to reduce churn on busy grids.

5. "Request Ride" workflow

  1. Customer requests a ride.
  2. An Aggregator server asks the QuadTree servers for nearby drivers.
  3. The aggregator collects results and sorts by rating.
  4. It notifies the top three drivers simultaneously; first to accept gets the ride, others receive a cancellation; no response → the next three.
  5. The customer is notified once a driver accepts.

6. Fault tolerance & ranking

Replicate Driver Location and Notification servers (secondary takes over); also persist this data on fast SSDs so even a double failure is recoverable. Top-rated drivers: store each driver's aggregate rating in DB and QuadTree; partitions return their top 10 by rating, the aggregator picks the overall top 10.

7. Advanced discussion points

  • Handling clients on slow, flaky, disconnecting networks.
  • A client disconnecting mid-ride — how does billing work?
  • What changes if clients pull everything instead of servers pushing?

Chapter 15 · Designing Ticketmaster

Hard Similar: bookmyshow.com — online movie ticket booking

1. Requirements

  • Functional: list cities with affiliate cinemas; show movies per city; show cinemas and showtimes per movie; book tickets for a show; display the hall's seating map; distinguish available vs booked seats; hold seats for five minutes pending payment; let users wait for seats that may free up when other holds expire; serve waiting users first-come-first-serve.
  • Non-functional: highly concurrent — many requests for the same seats must be handled gracefully and fairly; financial transactions demand security and an ACID-compliant database.

Assumptions: no user authentication (simplicity); no partial orders — all requested tickets or none; fairness is mandatory; cap 10 seats per booking to deter abuse; expect huge spikes on hot releases — must scale and stay available.

2. Capacity estimation

3B page views/month; 10M tickets sold/month. Storage: 500 cities × 10 cinemas × 2,000 seats × 2 shows/day × (50 B booking + 50 B movie/cinema info) = 2 GB/day → five years ≈ 3.6 TB.

3. APIs

SearchMovies(api_dev_key, keyword, city, lat_long, radius, start_datetime, end_datetime,
             postal_code, includeSpellcheck, results_per_page, sorting_order)
  → JSON: movie/show info incl. seat types ("Regular" $14.99, "Premium" $24.99) and fill status
ReserveSeats(api_dev_key, session_id, movie_id, show_id, seats_to_reserve[])
  → "Reservation Successful" | "Failed — Show Full" | "Failed — Retry, seats held by others"

The session_id ties the hold to the user's session so expiry can release it.

4. Database design

Each city has cinemas; each cinema has halls; each movie has shows; each show has bookings; users can have many bookings. Tables: City, Cinema(FK CityID), Cinema_Hall(FK CinemaID), Cinema_Seat(FK CinemaHallID), Movie, Show(FK CinemaHallID, MovieID), Show_Seat(FK CinemaSeatID, ShowID, BookingID; Status; Price), Booking(FK UserID, ShowID; NumberOfSeats; Timestamp; Status enum), User, Payment(FK BookingID; amount, timestamp, discount coupon, remote txn ID, method).

5. Booking workflow

Search → pick movie → pick show → choose seat count → if enough seats: pick seats on the hall map → system tries to reserve. If it can't: the show may be full (error); the chosen seats may be gone but others remain (back to the map); or all seats are either booked or held in the reservation pool — then the user enters a waiting page until seats free up, with these outcomes: enough seats free → seat map; everything gets booked or too few remain → error; user cancels; or the wait hits the one-hour cap and the session expires. After a successful reservation, the user has five minutes to pay; otherwise the hold is released to others.

6. The two daemon services

ActiveReservationsService

Per show, keep reservations in a LinkedHashMap (jump to any booking to remove it on completion; the head is always the oldest, so expiry pops from the front), inside a hashtable ShowID → {(BookingID, Timestamp)…}. In the DB, Booking.Status runs Reserved(1) → Booked(2), or Expired(3)/deleted on timeout, mirrored by removal from memory. This service also talks to the external financial service, and signals WaitingUsersService whenever a booking completes or expires.

WaitingUsersService

Same structure — ShowID → LinkedHashMap of (UserID, wait-start-time): jump-removal when users cancel, and the head is the longest-waiting user, guaranteeing first-come-first-serve. Clients hold long-poll requests for status updates; when seats free, the server answers the poll. Expiry buffer: the client shows a countdown that may drift from the server's — add five seconds server-side so the client never times out after the server and loses a purchase it thought it made.

7. Concurrency — no double-booked seats

Use SQL transactions with Serializable isolation — the highest level, safe from dirty, non-repeatable, and phantom reads; reading rows inside the transaction write-locks them:

SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
BEGIN TRANSACTION;
  -- reserve seats 54, 55, 56 for ShowID = 99
  SELECT * FROM Show_Seat WHERE ShowID = 99 AND ShowSeatID IN (54,55,56) AND Status = 0; -- free
  -- if exactly three rows returned: UPDATE Show_Seat ...; UPDATE Booking ...;  else fail
COMMIT TRANSACTION;

Only after the transaction commits does the reservation enter ActiveReservationsService.

8. Fault tolerance & partitioning

If ActiveReservationsService crashes, rebuild from the Booking table's Reserved(1) rows (or run master-slave). Waiting users exist only in memory — a crash loses them unless there's a master-slave pair. Databases: master-slave. Partitioning: by MovieID all shows of a hot movie hit one server — partition by ShowID instead. Both daemon services are allocated to app servers by consistent hashing on ShowID (say three servers per show). On expiry, the owning server: (1) updates the DB (remove/expire booking, free seats in Show_Seat), (2) removes it from the LinkedHashMap, (3) notifies the user, (4) broadcasts to all WaitingUserService servers for that show to find the longest-waiting user, (5) messages the server holding that user to proceed if enough seats freed. On a successful booking: notify all waiting-user servers so they expire anyone needing more seats than remain; those servers query the DB once (a DB cache helps) and iterate their LinkedHashMaps to prune.

Ticketmaster is the course's ACID showcase: everywhere else we happily traded consistency for availability; here money changes hands, so we reach for serializable transactions and accept the cost.

Part III · System Design Basics (Glossary)

Chapter 16 · Key Characteristics of Distributed Systems

Concept

Five qualities define a distributed system: scalability, reliability, availability, efficiency, and manageability.

Scalability

The capability to grow and manage increased demand — more data, more transactions — ideally without performance loss. In practice performance declines with size (network distance grows; some tasks resist distribution by their atomic nature or design flaws, capping speed-up); a scalable architecture balances load evenly across nodes. Horizontal scaling adds servers to the pool (easy to do dynamically — Cassandra, MongoDB); vertical scaling adds CPU/RAM/storage to one server (MySQL's typical path — but bounded, and often requiring downtime).

Reliability

The probability the system keeps delivering service even as components fail — any failing machine is replaced by a healthy one. Amazon's canon: a user transaction (a shopping cart!) is never lost to a machine failure — achieved through redundancy of components and data, which costs money; that's the price of eliminating single points of failure.

Availability

The percentage of time the system is operational. An aircraft that flies many hours a month without downtime is highly available; when it's down for maintenance it's not. Reliability vs. availability: reliable ⇒ available, but not conversely — a system can achieve high availability with minimal repair time and ready spares while still being unreliable. An online store might post 99.99% availability for two years, yet never having done security testing, suffer incident-driven outages (and reputational damage) in year three.

Efficiency

Two standard measures: latency (response time — delay to the first result) and throughput (bandwidth — items delivered per unit time), analyzed via number of messages and message sizes. Message-count analysis is admittedly simplistic — it ignores topology, load variation, and hardware/software heterogeneity — but precise cost models are impractical, so we live with rough, robust estimates.

Serviceability / Manageability

How easily the system is repaired and operated — time-to-fix directly hurts availability. Consider ease of diagnosis, ease of updates, and simplicity of operation; early fault detection (even systems that phone the service center automatically) shrinks downtime.

Chapter 17 · Load Balancing

Concept

A load balancer (LB) spreads traffic across a cluster, improving responsiveness and availability, and tracks resource health — servers that are down, unresponsive, or error-prone stop receiving traffic. Place LBs at three layers: user↔web servers, web↔internal platform (app or cache servers), and platform↔database.

Benefits

Faster, uninterrupted service (requests route around busy servers); less downtime and more throughput for providers; easier administration; smart LBs offer predictive analytics on traffic bottlenecks; and work is spread over many devices instead of stressing one.

Health checks & algorithms

LBs forward only to healthy servers — regular health checks connect to each backend and pull failures from the pool until they respond again. Selection algorithms: Least Connection (fewest active connections — good for uneven persistent connections), Least Response Time (fewest connections + lowest average response time), Least Bandwidth (least Mbps currently served), Round Robin (cycle through the list — best for equal servers and short-lived connections), Weighted Round Robin (capacity-weighted rotation), and IP Hash (hash the client's IP to pick the server).

Redundant LBs

The LB itself can be the single point of failure — cluster two, each monitoring the other; the passive takes over when the active fails.

Chapter 18 · Caching

Concept

Load balancing scales you out; caching makes vastly better use of what you already have, exploiting locality of reference — recently requested data is likely to be requested again. Like short-term memory: small, fast, holding the most recent items; caches appear at every layer (hardware, OS, browsers, web apps) and work best near the front end.

Application server cache & CDN

A cache on a request node returns local data instantly (memory first, then local disk — both beat network storage). With many nodes behind a random LB, the same request hits different nodes and misses grow — solved by global or distributed caches. A CDN serves large static media: it returns content if cached locally, otherwise fetches from your backend, caches, and serves. Not big enough for a CDN yet? Serve static media from a separate subdomain (static.yourservice.com) on lightweight Nginx and flip DNS to a CDN later.

Cache invalidation — three write policies

  • Write-through: write to cache and database simultaneously — full consistency, crash-safe, but every write costs double latency.
  • Write-around: write straight to storage, bypassing cache — avoids flooding the cache with never-reread data, but fresh writes always miss on first read.
  • Write-back: write to cache only, confirm immediately, flush later — lowest latency, highest throughput, but data loss risk while the only copy sits in cache.

Eviction policies

FIFO (evict oldest-arrived), LIFO (evict newest-arrived), LRU (evict least recently used — the course's workhorse), MRU (evict most recently used), LFU (evict least frequently used), and Random Replacement.

Chapter 19 · Sharding / Data Partitioning

Concept

Partitioning breaks a big database into smaller parts across machines, improving manageability, performance, availability, and load balance — past a certain scale, adding machines beats buying beefier ones.

Partitioning methods

  • Horizontal (range-based): different rows in different tables — e.g., places with ZIP < 10000 here, ≥ 10000 there. Careless range choices unbalance servers (Manhattan vs. its suburbs).
  • Vertical: feature-specific tables on their own servers (profiles here, friend lists there, photos on a third). Straightforward, low application impact — until one feature outgrows a single server (10B photos × 140M users) and must be re-partitioned anyway.
  • Directory-based: a lookup service maps each tuple key to its DB server, hiding the scheme from application code — servers can be added or the scheme changed without touching the app.

Partitioning criteria

  • Key/hash-based: hash a key attribute → partition (e.g., ID % 100). Uniform, but effectively fixes the server count — adding servers means rehashing and downtime; the workaround is consistent hashing.
  • List: each partition owns a value list (Nordic-country users together).
  • Round-robin: tuple i → partition (i mod n).
  • Composite: combine schemes — consistent hashing is essentially hash + list partitioning.

Common problems

Operations spanning tables or rows now cross servers: joins become impractical (denormalize and accept inconsistency risks); referential integrity — cross-shard foreign keys are mostly unsupported, so apps enforce it in code and run SQL cleanup jobs for dangling references; rebalancing — uneven data or load forces scheme changes and wholesale data movement, extremely hard without downtime; directory-based partitioning eases it at the cost of complexity and a new single point of failure (the lookup service).

Chapter 20 · Indexes

Concept

Indexing is the first tool to reach for when database performance sags. An index over one or more columns speeds up random lookups and ordered access. Mental model: a library catalog — one copy sorted by title, one by writer, each pointing to where the book lives; likewise an index stores the indexed column plus a pointer to the full row. For terabyte-scale datasets with tiny payloads (~1 KB), spread over many physical devices, indexes aren't optional — they're the only way to find the right physical location in reasonable time.

The write penalty

An index speeds reads but must itself be updated on every insert, update, and delete — and it consumes space. Never add indexes casually; remove unused ones. If a table is written often and read rarely, indexing likely costs more than it earns.

Chapter 21 · Proxies

Concept

A proxy is an intermediary between clients and back-end servers, used to filter, log, or transform requests (headers, encryption, compression) — and its cache can answer many requests without touching the remote server when multiple clients want the same resource.

Types

  • Open proxy: usable by any Internet user (unlike closed, group-only proxies). Two famous kinds: the anonymous proxy reveals itself as a server but hides the client's original IP; the transparent proxy identifies itself and exposes the original IP via HTTP headers — its main benefit is website caching.
  • Reverse proxy: retrieves resources from one or more servers on the client's behalf, returning them as if they originated from the proxy itself.

Chapter 22 · Redundancy and Replication

Concept

Redundancy duplicates critical components or functions to raise reliability — a backup or fail-safe (one copy of a file on one server means one server loss loses the file) — and removes single points of failure: with two service instances in production, failure of one triggers failover to the other. Replication shares information to keep redundant resources consistent, improving reliability, fault-tolerance, and accessibility. In DBMSs it's usually master-slave: the master takes all updates, which ripple to the slaves; each slave acknowledges receipt so subsequent updates can flow.

Chapter 23 · SQL vs. NoSQL

Concept

Relational databases are structured with predefined schemas — like a phone book; non-relational databases are unstructured and distributed with dynamic schemas — like a file folder holding everything from addresses to likes.

The families

SQL: rows and columns — MySQL, Oracle, MS SQL Server, SQLite, Postgres, MariaDB. NoSQL: key-value stores (Redis, Voldemort, Dynamo); document databases (CouchDB, MongoDB — documents grouped in collections, each free-form); wide-column stores (Cassandra, HBase — column families instead of tables, columns per row can vary, great for big analytics); graph databases (Neo4J, InfiniteGraph — nodes, properties, and edges for graph-shaped data).

The differences

SQLNoSQL
StorageTables: row = entity, column = data pointKey-value, document, graph, or columnar models
SchemaFixed; every row conforms; altering means going offlineDynamic; add columns on the fly; rows needn't fill every column
QueryingSQL — very powerfulUnQL, collection-focused, syntax varies per store
ScalabilityVertical (expensive); cross-server scaling is hard and slowHorizontal on cheap commodity/cloud hardware, often auto-distributed
ACIDVast majority fully compliantMostly sacrificed for performance and scale

Choosing

Pick SQL when you need ACID (e-commerce, finance — it prescribes exactly how transactions interact and protects integrity) or when data is structured, unchanging, and growth is modest. Pick NoSQL to store large volumes of loosely-structured data without predefined types; to exploit cloud economics (commodity hardware; Cassandra scales across data centers out of the box); or for rapid development with frequently changing data structures, where a relational schema-migration cycle would slow you down. Many businesses run both.

Chapter 24 · CAP Theorem

Concept

No distributed system can simultaneously guarantee more than two of: Consistency — all nodes see the same data at the same time (achieved by updating several nodes before further reads); Availability — every request receives a success/failure response (achieved by replicating across servers); Partition tolerance — the system keeps working despite message loss or partial failure, surviving any network failure short of total collapse, because data is replicated across node and network combinations.

Why the trade-off is forced: consistency demands all nodes see the same updates in the same order; under a network partition, updates on one side may not reach the other before a client reads stale data there. The only remedy is to stop serving from the out-of-date partition — surrendering 100% availability. Trading off among CAP is practically the first decision in any distributed design.

Chapter 25 · Consistent Hashing

Concept

Distributed Hash Tables map keys to storage locations via index = hash(key). The intuitive key % n over n cache servers has two fatal flaws: it's not horizontally scalable (adding one host breaks all existing mappings — scheduling downtime to remap everything is a maintenance nightmare) and it may not balance load (real data is non-uniform: some caches run hot and saturated while others idle).

The ring

Consistent hashing minimizes reorganization when nodes come and go: on resize, only k/n keys remap (k keys, n servers) versus all of them under mod-hashing. Mechanics: map the hash output range (say [0, 256)) onto a ring; hash each cache server onto the ring; to place a key, hash it and walk clockwise to the first cache encountered. Adding server D splits only the keys that lived on its clockwise successor; removing (or losing) server A moves only A's keys to the next server — nothing else shifts.

Virtual replicas

Random data still lands unevenly, so map each cache to multiple ring points ("virtual replicas"); with a well-mixing hash function, more replicas mean better key balance.

Chapter 26 · Long-Polling vs. WebSockets vs. Server-Sent Events

Concept

Baseline HTTP: the client opens a connection and asks; the server computes and responds on that request. Three patterns deliver more real-time behavior:

Ajax polling

The client asks at fixed intervals (e.g., every 0.5 s); no data means an empty response. Simple, but most responses are empty — pure HTTP overhead.

HTTP long-polling ("Hanging GET")

The client asks, expecting the server might not answer immediately; the server holds the request until data exists or a timeout fires, then sends a full response; the client immediately re-requests, so the server almost always has a waiting request to deliver events into. Each long-poll has a timeout, so clients reconnect periodically.

WebSockets

Full-duplex communication over a single TCP connection, established by the WebSocket handshake: afterwards both sides send data at any time, with low overhead — a standardized way for the server to push unasked, enabling a true two-way ongoing conversation.

Server-Sent Events (SSE)

The client opens one persistent, long-term connection over regular HTTP; the server pushes whenever news arrives; client→server traffic needs a different channel. Best when the server generates a stream of events toward the client.

Map them back to the case studies: Messenger and the newsfeed lean on long polling; Dropbox's sync notifications, Ticketmaster's waiting page, and Uber's location fan-out all pick from this same menu. Choosing the transport is a design decision — say so in the interview.
