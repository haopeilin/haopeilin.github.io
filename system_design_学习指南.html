CHAPTER 01

分布式系统的核心特性Key Characteristics of Distributed Systems

设计大型系统需思考：有哪些架构组件（Architectural Pieces）？它们如何协同工作？如何做出正确的权衡（Tradeoffs）？虽然过早投资扩展通常不明智，但提前进行架构思考可以节省大量时间和资源。

📈 可扩展性 Scalability

系统在增长的工作量（更多数据、更多事务）下持续运转的能力。实际中由于管理开销（如网络延迟），性能通常随规模下降。某些任务具有原子性无法被分布执行，最终会限制分布式加速效果。可扩展的架构会尝试均匀地平衡所有节点的负载。

Vertical vs Horizontal Scaling — 纵向有上限且需停机，横向更灵活
🛡️ 可靠性 Reliability

系统在给定时间段内不发生故障的概率。可靠的分布式系统即使部分软硬件故障仍能服务。关键手段是冗余（Redundancy）——软件组件和数据都备份，消除每一个单点故障（Single Point of Failure）。

💡 实战案例

Amazon 购物车的可靠性

用户加入购物车后服务器宕机，另一台持有精确副本（Replica）的服务器立即接管，用户无感知。冗余有成本，但可靠系统必须为此付出代价。

⏱️ 可用性 Availability

系统在特定时间段内保持正常运行的百分比。考虑可维护性、修复时间、备件可用性等因素。一架每月能飞行很多小时且停机很少的飞机拥有高可用性。

⚠️ Reliability vs Availability

可靠性 vs 可用性

可靠 → 一定可用；但可用 ≠ 一定可靠。一个在线零售系统前两年有 99.99% 可用性，但未做信息安全测试。第三年遭遇安全事件导致长时间不可用，给客户造成声誉和财务损失。高可用性可以通过缩短修复时间和确保备件随时可用来实现，即使系统不够可靠。

⚡ 效率 Efficiency

两个标准：响应时间/延迟（Latency）——获取第一个结果的延迟；吞吐量/带宽（Throughput）——单位时间内交付的结果数。对应两种单位成本：

① 系统节点间全局发送的消息数量（Number of messages），不论消息大小

② 消息的大小（Size of messages），代表数据交换的总量

仅用"消息数量"分析过于简化——忽略了网络拓扑、负载变化、软硬件异构性等。但精确的成本模型很难构建，我们只能使用粗略但稳健的估算。

🔧 可维护性 Serviceability / Manageability

系统维修和维护的简便程度。修复时间越长，可用性越低。需要考虑：诊断和理解问题的难易度、更新修改的难易度、日常运行是否顺畅。

早期故障检测可减少或避免停机。例如一些企业系统在检测到故障时能自动拨打服务中心（无需人工干预）进行报告。

CHAPTER 02

负载均衡Load Balancing

负载均衡器（LB）将流量分散到一组服务器上，提升响应速度和可用性。LB 跟踪所有资源状态——如果服务器不可用、无响应或错误率过高，LB 停止向其发送流量。LB 位于客户端和服务器之间，使用各种算法将请求分配到多台后端服务器，防止任何单一服务器成为单点故障。

可以在系统每一层添加 LB：① 用户↔Web 服务器 ② Web 服务器↔应用/缓存服务器 ③ 内部平台层↔数据库

✅ 负载均衡的好处 Benefits of Load Balancing
🚀

用户体验更快更流畅

请求不会卡在繁忙服务器上等待，而是被立即转发到更空闲的资源。

🛡️

更高吞吐更少停机

即使服务器完全宕机，LB 也会自动绕过路由到健康服务器，终端用户无感知。

⚙️

运维更轻松

负载分散使管理员更容易处理入站请求，同时降低用户等待时间。

🔮

智能预测分析 Predictive Analytics

智能 LB 能通过预测分析在流量瓶颈发生前识别它们，为自动化和业务决策提供可行洞察。

💚

更少的组件压力

工作由多台设备分担，每台只承担一小部分，减少单个设备的故障和压力。

🔄 负载均衡算法 Load Balancing Algorithms

LB 转发请求前：① 确认目标服务器在正常响应 ② 使用预配置算法从健康服务器中选择一台。

🏥 健康检查 Health Checks

LB 如何知道服务器是否健康？

LB 定期尝试连接后端服务器进行"健康检查"。未通过检查的服务器被自动从池中移除，流量不再转发给它，直到再次通过健康检查。

📉 最少连接法 Least Connection Method▼
将流量导向当前活跃连接数最少的服务器。当大量持久客户端连接在服务器间分布不均时特别有效。
⚡ 最短响应时间法 Least Response Time Method▼
综合：最少活跃连接数 + 最低平均响应时间。"最少连接法"的增强版。
📶 最少带宽法 Least Bandwidth Method▼
选择当前正在提供最少流量（以 Mbps 计）的服务器。适用于带宽敏感的场景。
🔄 轮询法 Round Robin Method▼
按顺序轮流分配请求。到达列表末尾后从头开始。适用于服务器性能相近且没有大量持久连接的场景。
⚖️ 加权轮询法 Weighted Round Robin Method▼
为不同处理能力的服务器分配不同权重（Weight，整数值表示处理能力）。权重高的服务器优先获得新连接，也获得更多连接总量。
🔐 IP 哈希法 IP Hash▼
对客户端 IP 地址哈希计算后路由到对应服务器。同一客户端请求始终到达同一服务器，利于 Session 保持。
🔁 冗余负载均衡器 Redundant Load Balancers

LB 本身也可能成为单点故障。解决方案：第二台 LB 与第一台组成集群（Cluster）。两台 LB 互相监控健康状态（heartbeat），两者都具备同等的流量服务和故障检测能力。主 LB 故障时，备用 LB 立即接管（Failover）。

CHAPTER 03

缓存Caching

负载均衡帮助横向扩展，而缓存让你更高效利用已有资源。缓存利用引用局部性原理（Locality of Reference）：最近被请求的数据很可能再次被请求。缓存像短期记忆——空间有限但比原始数据源快得多。缓存存在于计算的几乎所有层级：硬件、操作系统、Web 浏览器、Web 应用等。缓存通常靠近前端，能快速返回数据而不给下游增加负担。

💻 应用服务器缓存 Application Server Cache

在请求层节点直接放置缓存，允许本地存储响应数据。命中则快速返回，未命中则从磁盘查询。缓存可在内存中（极快）或本地磁盘上（比网络存储快）。

扩展到多个节点时：LB 随机分配请求导致同一请求落在不同节点上，Cache Miss 增加。解决方案：

① 全局缓存（Global Cache）——所有节点共享一个缓存层

② 分布式缓存（Distributed Cache）——缓存本身也被分片到多个节点

🌐 CDN — 内容分发网络 Content Distribution Network

CDN 是针对大量静态媒体的缓存。请求先到 CDN，有本地缓存就直接返回；没有则查询后端，缓存后再返回。
小技巧：系统不够大时先用独立子域名（如 static.yourservice.com）配合 Nginx 提供静态资源，未来轻松切换 DNS 到 CDN。

🔄 缓存失效 Cache Invalidation

缓存需维护与数据源的一致性（Coherence）。数据在数据库被修改后，缓存应被失效；否则导致不一致。

直写缓存 Write-Through

数据同时写入缓存和数据库。完全一致，崩溃/断电不丢数据。

⚠️ 每次写执行两次才返回成功，写入延迟较高。

绕写缓存 Write-Around

数据直接写入永久存储，绕过缓存。减少缓存被不常读取的写操作淹没。

⚠️ 刚写入的数据被读取时产生 Cache Miss，需从更慢的后端读取。

回写缓存 Write-Back

数据仅写入缓存，立即确认完成。之后按间隔或条件写入永久存储。

✅ 低延迟 + 高吞吐，适合写密集型应用。

⚠️ 崩溃时可能丢失数据（唯一副本只在缓存中）。

🗑️ 缓存淘汰策略 Cache Eviction Policies
策略英文名规则

LIFOLast In First Out最后进入缓存的最先淘汰，不考虑访问频率

CHAPTER 04

数据分区 / 分片Sharding / Data Partitioning

将大型数据库拆分为多个更小部分，提升可管理性、性能、可用性和负载均衡。超过一定规模后，横向添加更多机器比纵向升级单台服务器更经济可行。

📦 分区方法 Partitioning Methods
↔️

水平分区

Horizontal Partitioning / Range-based Sharding

不同的行（Rows）放入不同表。例如 ZIP < 10000 存表A，≥10000 存表B。
⚠️ 分区键选择不当导致不均匀——曼哈顿数据远多于郊区。

↕️

垂直分区

Vertical Partitioning

按功能特性分割。如 Instagram：用户资料→A，好友→B，照片→C。实现简单，影响小。
⚠️ 单功能增长后（140M 用户 10B 照片）单服务器无法承载。

📖

基于目录的分区

Directory-Based Partitioning

创建查找服务（Lookup Service）维护映射。松耦合：添加服务器或改分区方案对应用透明。
⚠️ 增加复杂度，查找服务成新单点故障。

📐 分区标准 Partitioning Criteria
🔑 键/哈希分区 Key or Hash-based Partitioning▼
对键属性应用哈希函数得到分区编号。例如 100 台 DB 服务器，ID % 100。确保数据均匀分配。根本问题：固定了 DB 服务器总数，添加新服务器需改哈希函数、重新分配数据并停机。解决方案：一致性哈希（Consistent Hashing）。
📋 列表分区 List Partitioning▼
每个分区分配一组值列表。例如冰岛、挪威、瑞典、芬兰、丹麦用户存储在"北欧国家"分区中。
🔄 轮询分区 Round-Robin Partitioning▼
最简单策略，确保均匀数据分布。有 n 个分区时，第 i 个元组分配到分区 i mod n。
🧩 组合分区 Composite Partitioning▼
组合上述方案。如先列表分区再哈希分区。一致性哈希可视为哈希+列表分区的组合——哈希将键空间缩减到可列举的大小。
⚠️ 分片常见问题 Common Problems of Sharding
🔗 跨分片连接与反规范化 Joins & Denormalization▼
跨分片 JOIN 需从多台服务器编译数据，低效或不可行。常见方案：反规范化（Denormalize）将需 JOIN 的数据合并到单表。代价：处理数据不一致。
🔒 引用完整性 Referential Integrity▼
大多数 RDBMS 不支持跨库外键约束。应用需在代码层面强制执行，并经常运行 SQL 清理任务处理悬挂引用（Dangling References）。
⚖️ 再平衡 Rebalancing▼
原因：① 数据分布不均（某 ZIP Code 数据量无法放入单分区）② 某分片负载过高（用户照片分片请求太多）。需创建更多分片或重新调整，所有数据迁移。不停机完成极其困难。基于目录的分区可让再平衡更易管理，但增加复杂度并引入新单点故障。
CHAPTER 05

数据库索引Indexes

当数据库性能不再满意时，索引是首先考虑的优化。目标：让搜索表中特定行变得更快。索引可基于一个或多个列创建，支持快速随机查找和有序记录高效访问。

📚 类比 — 图书馆目录 Library Catalog

索引就像图书馆的目录卡片

图书馆目录通常有四列：书名、作者、主题、出版日期。通常有两个目录：一个按书名排序，一个按作者排序。可以按作者查（想读某作者的书），也可按书名查（知道书名不知作者）。这些目录提供了按相关信息排序的、易于搜索的数据列表——这就是索引的本质。

索引是一种数据结构，可理解为"目录（Table of Contents）"指向实际数据位置。为某列创建索引时，存储该列值和指向完整行的指针。

对于数据量达 TB 级但有效负载很小（如 1 KB）的大数据集，索引是优化数据访问的必需品——不可能遍历如此庞大的数据，且数据很可能分布在多个物理设备上。

⚖️ How Indexes Decrease Write Performance

索引的代价：写入性能下降

索引能极大加速读取，但索引本身可能很大（额外的键）。每次 INSERT、UPDATE、DELETE 不仅要写数据还要更新索引。应避免添加不必要的索引，移除不再使用的索引。如果数据库主要用于写入且很少读取，索引代价可能不值得。

CHAPTER 06

代理服务器Proxies

代理是客户端和后端服务器之间的中间层（Intermediary）。客户端连接代理请求服务（网页、文件、连接等）。代理可过滤、记录日志、转换请求（添加/移除 Header、加密/解密、压缩）。代理缓存还能为大量重复请求服务，无需每次访问远程服务器。

🌐 开放代理 Open Proxy

任何互联网用户都可访问的代理。一般代理只允许网络组内用户使用（封闭代理 Closed Proxy），用于存储和转发 DNS 或网页等互联网服务，减少和控制组内带宽使用。开放代理允许任何人使用。两种常见类型：

🎭

匿名代理 Anonymous Proxy

表明自己是代理服务器，但不透露客户端原始 IP 地址。容易被发现，但对需要隐藏 IP 的用户有价值。

👁️

透明代理 Transparent Proxy

表明自己是代理，通过 HTTP Header 可查看客户端原始 IP。主要优势是缓存网站内容的能力。

🔄 反向代理 Reverse Proxy

代表客户端从一个或多个后端服务器获取资源，然后返回给客户端。看起来像是资源直接来自代理服务器本身。与正向代理（代表客户端）不同，反向代理代表的是服务器端。

CHAPTER 07

冗余与复制Redundancy and Replication

📋 冗余 Redundancy

对系统关键组件或功能的复制（Duplication），目的是提高可靠性（作为备份/故障安全 Fail-safe）或提升性能。如果文件只存储在单台服务器上，该服务器丢失=文件丢失。通过创建冗余副本解决。冗余在消除单点故障中起关键作用。例如两个服务实例在生产环境运行，一个挂了，系统可以故障转移（Failover）到另一个。

🔄 复制 Replication

在冗余资源之间共享信息以保持一致性，提高可靠性、容错性或可访问性。在许多 DBMS 中广泛使用，通常采用主从关系（Master-Slave Relationship）：

① 主节点（Master）接收所有更新

② 更新传播（Ripple Through）到所有从节点（Slaves）

③ 每个从节点发回确认消息（ACK）表示已成功接收更新，从而允许发送后续更新

SQL vs NoSQLRelational vs Non-relational Databases

两大阵营：SQL（关系型）和NoSQL（非关系型）。关系型有预定义模式（像电话簿）；非关系型非结构化、分布式、动态模式（像文件夹，包含一切）。

维度SQLNoSQL
存储行列表格。每行=实体，每列=数据点（Color, Make, Model）多种模型：键值对、文档、图、宽列
模式固定模式（Fixed Schema），修改需全库离线动态模式（Dynamic Schema），列可随时添加
查询SQL（Structured Query Language），功能强大各库自有语法，称为 UnQL
扩展纵向为主（更强硬件，成本高），跨服务器扩展困难耗时横向为主，廉价硬件或云实例即可，许多技术自动分发数据
ACID✅ 绝大多数 ACID 兼容⚠️ 多数牺牲 ACID换性能和扩展性
🗃️ NoSQL 四大类型
🔑

键值存储

Key-Value Stores

键值对数组。代表：Redis, Voldemort, Dynamo

📄

文档数据库

Document Databases

数据存储在文档中，每个文档结构可完全不同。代表：CouchDB, MongoDB

📊

宽列数据库

Wide-Column Databases

"列族"代替"表"，不需预知所有列，每行列数可不同。适合大数据分析。代表：Cassandra, HBase

🕸️

图数据库

Graph Databases

节点（实体）、属性（信息）、边（连接）。代表：Neo4J, InfiniteGraph

🤔 如何选择？

选择 SQL 当…

① 需要 ACID 合规——减少异常、保护数据完整性（电商/金融首选）
② 数据结构化且不经常变化，无大规模增长需求

选择 NoSQL 当…

① 存储海量非结构化数据——无需预定义类型，随需添加
② 充分利用云计算和存储——数据轻松分布到多台服务器（如 Cassandra 设计时就考虑了跨数据中心扩展，无需大量额外配置）
③ 快速迭代开发——不需提前准备数据结构，频繁更新不会导致大量停机

CHAPTER 09

CAP 定理CAP Theorem

分布式软件系统不可能同时提供三个保证的全部。设计分布式系统时，CAP 权衡几乎是首先要考虑的事。只能选择其中两个：

🔄

一致性 Consistency

所有节点同一时间看到相同数据。通过在允许读取前更新多个节点实现。

✅

可用性 Availability

每个请求都收到成功/失败响应。通过在不同服务器间复制数据实现。

🌐

分区容错 Partition Tolerance

系统在消息丢失或部分故障时继续工作。数据充分复制以应对间歇性中断。

CA: RDBMS — CP: BigTable, MongoDB, HBase — AP: Cassandra, CouchDB

为什么不可能三者兼得？一致性要求所有节点以相同顺序看到更新。但网络分区时，一个分区的更新可能无法到达另一个分区。客户端先从已更新分区读取、再从未更新分区读取就会看到不一致数据。唯一办法是停止从过时分区提供服务——但这样服务不再 100% 可用。

CHAPTER 10

一致性哈希Consistent Hashing

📋 背景：分布式哈希表 Distributed Hash Table (DHT)

DHT 是分布式可扩展系统的基础组件。哈希表需要键、值和哈希函数：index = hash_function(key)。

假设设计分布式缓存，有 n 台缓存服务器。直觉哈希函数 key % n 简单常用，但有两个主要缺陷：

① 不可横向扩展（NOT horizontally scalable）——添加新缓存主机时所有已有映射失效，有大量数据时难以安排停机更新

② 可能导致负载不均（NOT load balanced）——数据分布不均匀时，某些缓存过热饱和，其他缓存几乎空闲

💡 什么是一致性哈希？

对分布式缓存和 DHT 非常有用的策略。以最小化重组方式分布数据。调整大小时只需重新映射 k/n 个键（k=总键数，n=总服务器数），而非全部。

原则：对象尽可能映射到相同主机。主机移除→其对象被其他主机分担；新主机加入→从少数主机接管部分键。

⚙️ 工作原理 How does it work?

假设哈希输出范围 [0, 256)，将整数想象排列在一个环上：

① 将所有缓存服务器哈希到环上的位置

② 映射键到服务器：将键哈希为整数，沿顺时针移动，遇到的第一个缓存就是存储位置

添加新服务器（如 D）：原在 C 上的部分键分给 D，其他不受影响。移除缓存（如 A）：原映射到 A 的键落到下一个（B），只需迁移这些键。

数据分布可能不均匀。解决：为每台缓存在环上创建多个虚拟副本（Virtual Replicas），使每台缓存关联多个区段。哈希函数混合均匀时，虚拟副本越多键分布越均衡。

CHAPTER 11

通信协议Long-Polling vs WebSockets vs SSE

先理解标准 HTTP 请求：① 客户端打开连接请求数据 → ② 服务器计算响应 → ③ 服务器在已开连接上发回响应。

Ajax 轮询 Ajax Polling

AJAX 应用的标准技术。客户端反复请求服务器。① 用常规 HTTP 请求 → ② 页面按固定间隔（如 0.5s）发送请求 → ③ 服务器像常规 HTTP 计算并返回 → ④ 客户端周期性重复。

⚠️ 客户端必须不断请求，大量空响应造成 HTTP 开销浪费。

HTTP 长轮询 Long-Polling / "Hanging GET"

服务器可能不会立即响应。① 客户端发请求并等待 → ② 服务器延迟响应直到有更新或超时 → ③ 有更新时发送完整响应 → ④ 客户端收到后立即（或短暂延迟）再次发起长轮询 → ⑤ 每个请求有超时，超时后需重连。

WebSocket

通过单个 TCP 连接提供全双工（Full Duplex）通信。持久连接，双方任何时刻可发数据。通过 WebSocket 握手建立。允许服务器无需客户端请求就推送内容，消息在连接保持打开时来回传递——真正的双向（Bi-directional）持续对话。

✅ 最低开销的双向实时通信。

服务端推送事件 Server-Sent Events (SSEs)

客户端建立持久长期连接。服务器用此连接向客户端发数据。客户端若要发数据需用其他协议。① 客户端用 HTTP 请求 → ② 页面打开到服务器的连接 → ③ 服务器有新信息时发数据。

✅ 适合服务器到客户端实时推送，或服务器在循环中生成数据发送多个事件。
